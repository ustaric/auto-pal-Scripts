# 팰월드 통합 관리 시스템 설치 및 운영 가이드

이 가이드는 구글 클라우드(GCP) VM 인스턴스를 생성하기 위한 사전 준비(계정, 디스코드 봇, SSH 키 발급) 과정부터 가상 서버에 처음 접속하여 게임 서버와 디스코드 관리 봇을 빌드하고 안정적으로 상시 구동하기까지의 전 과정을 상세하게 다룹니다.

---

## 📋 1. 시스템 구성 및 주요 개선 기능 요약

* 대상 시스템: Google Cloud Platform (GCP)
* 권장 가상 서버 OS: Ubuntu 22.04 LTS
* 기본 작업 디렉토리: `/home/<사용자_계정명>/palworld` (환경 변수 `BASE_PATH`로 지정)

### 주요 개선 및 기능 요약

* **환경 설정 일괄 변경 및 이중 승인 (`/설정변경`)**
    * 여러 환경 변수(`.env`) 값을 쉼표(`,`)로 구분하여 한 번에 변경합니다.
    * 입력된 키의 유효성을 검증하여 오타/누락 시 수정을 거부합니다.
    * 변경 전후 값을 비교하는 임베드와 [변경 승인] 버튼을 통해 안전한 이중 확인 절차를 제공합니다.
    * 설정 변경 공지를 포함한 모든 주요 시스템 알림 메시지는 15분(900초) 후 자동 삭제되어 채널 청결을 유지합니다.

* **서버 재시작 시 안전 백업 연동 (`/restart`)**
    * 관리자가 `/restart` 명령 실행 시, 서버 정지 직전 데이터를 안전하게 저장하고 백업(`run_backup`)을 선행한 후 컨테이너를 재기동합니다.

* **Docker Compose V2 호환성 및 안정성 확보**
    * 최신 도커 API 규격에 맞춰 Docker Compose V2 환경과의 호환성을 확보하고, 기존 구형 컴포즈의 크래시 오류를 해결했습니다.
    * 안전한 `subprocess.run` 호출 방식을 사용하여 쉘 인젝션 위험을 방지하고 경로 제어를 강화했습니다.

* **지능형 상태 검증 (STARTING ➔ ONLINE)**
    * 도커 컨테이너가 켜지더라도 실제 게임 REST API가 버전 정보를 반환하기 전까지는 대시보드 상태를 STARTING (로딩 중)으로 고정합니다.
    * `player_watcher_task`의 5초 주기 감시를 통해 로딩 완료 즉시 상태를 ONLINE으로 업데이트합니다.

* **지능형 퇴장 감시 예약 백업**
    * 동접자 0명 감지 시 15분(`EMPTY_BACKUP_DELAY`)간 대기 후 백업을 예약합니다.
    * 이 대기 시간 중 유저가 재접속하면 예약된 백업을 자동으로 취소하며, 15분간 0명 유지 시 최종 세이브 데이터 보호를 위해 백업을 실행합니다.

* **12시간 주기 디스크 잔량 모니터링 및 경고**
    * `scheduled_disk_check_task`를 통해 12시간마다 디스크 사용률을 체크합니다.
    * 사용률이 80% 이상일 경우 지정된 로그 채널로 경고 임베드 메시지를 전송하여 용량 관리를 돕습니다.

* **실시간 디스크 상태 조회 (`/디스크`)**
    * 12시간 대기 없이, `/디스크` 명령어로 현재 서버의 디스크 전체 용량, 사용량, 사용률을 실시간 임베드로 즉시 조회할 수 있습니다.

* **향상된 이중 동적 자동 백업**
    * 유저 접속률이 높은 피크타임(평일 19시~02시, 주말 09시~02시)에는 30분 간격, 비피크타임에는 1시간 간격으로 동적 자동 백업을 수행합니다. (접속자가 0명일 시 백업 파일 생성을 대기하여 시스템 리소스를 보존합니다.)

* **정제된 대시보드 UI**
    * 기가바이트 단위를 직관적으로 표기하는 RAM 인터페이스(`2.1GB/16GB`) 및 모바일 화면에서도 가로폭이 보존되도록 레이아웃을 구성했습니다.

* **전용 알림 시스템**
    * 백업 성공/실패, 정기 점검 재시작, 디스크 용량 경고 등의 상황을 디스코드 로그 전용 채널(`LOG_CHANNEL_ID`)로 실시간 포워딩합니다.

## 🔒 2. 보안 및 안전성 설계 (Security & Safety)

본 시스템은 외부 해킹 공격 및 허가되지 않은 사용자의 서버 오용을 방지하기 위해 다음과 같은 다중 보안 설계가 적용되어 있습니다.

* **1. 폐쇄형 내부망 통신 (Localhost Isolation)**
    * 봇(`main.py`)과 게임 서버(`docker-compose`)는 동일한 GCP 가상 서버 내부에 상주합니다.
    * 봇이 정보를 수집하는 REST API 포트(`8212`)와 원격 명령어 포트(`25575`)는 외부 인터넷 IP 주소가 아닌 로컬 루프백 주소(`127.0.0.1`)를 통해서만 내부적으로 연결됩니다.

* **2. 가상 서버 방화벽 최소화 (Port Protection)**
    * 게임 플레이를 위해 열어두어야 하는 게임 포트(`UDP 8211`) 외의 모든 제어 포트는 외부로부터 차단할 수 있습니다.
    * 구글 클라우드 방화벽 설정 시, RCON 포트(`25575`) 및 REST API 포트(`8212`)를 외부에 개방하지 않아도 내부 통신망을 통해 봇이 직접 정보를 제어하므로 외부로부터의 불필요한 포트 노출을 방지합니다.

* **3. 디스코드 관리자 ID 검증 구조 (Role Verification)**
    * 서버 제어 및 백업과 관련된 중요 슬래시 명령어는 호출한 디스코드 유저의 숫자 고유 ID와 `env`에 지정된 `ADMIN_IDS` 목록을 1대1로 대조하여 검증합니다.
    * 관리자로 등록되지 않은 일반 유저나 타인이 명령어를 사용하려 하더라도 봇이 즉각 거부(`❌ 권한이 없습니다.`)하도록 구현되어 있습니다.

* **4. 강화된 환경 변수 사전 검증 (Fail-Fast)**
    * 봇 기동 직후 `env` 및 `.env` 파일의 필수 설정값(토큰, 채널 ID, 경로, 포트 범위 등)을 검증합니다.
    * 유효하지 않은 값이 발견될 경우, 봇이 구동하기 전에 스스로를 안전하게 중단하여 잠재적 오류나 오작동을 사전에 차단합니다.

* **5. 안전한 Subprocess 실행 및 쉘 인젝션 방지**
    * 쉘 주입 취약점을 사전에 차단하기 위해 `os.system()` 대신 `subprocess.run()`에 리스트 인자 형식(`["docker", "exec", ...]`)을 사용합니다.
    * 디렉토리 이동(`cd ... && docker-compose ...`) 방식 대신 `cwd=BASE_PATH` 옵션을 활용하여 파이썬이 리눅스 디스크 디렉토리를 직접 가리키도록 정밀하게 지정했습니다.

* **6. 고급 로깅 및 예외 추적 (Traceback)**
    * `logging` 라이브러리의 `RotatingFileHandler`를 적용하여 `palworld_bot.log` 파일에 로그를 기록합니다. 파일 크기가 5MB에 도달하면 새 파일을 열고 기존 로그를 보존(최대 3개)하여 디스크 용량 오버플로우를 예방합니다.
    * 모든 `print()` 구문을 `logger.info()`, `logger.error()`, `logger.warning()` 등으로 교체했습니다.
    * 의의 불명확한 `except:` 처리 대신 `except Exception as e:`와 `logger.exception()`을 사용하여 예외 발생 시 상세한 에러 로그와 호출 스택(`traceback`)을 기록하도록 정비했습니다.

## 🔑 5. [사전 준비] 개발 환경 및 클라우드 사전 세팅 가이드

팰월드 서버를 원활히 구동하고 관리하기 위해 가상 서버 생성 이전에 아래의 세팅을 순서대로 진행해야 합니다.

### 1. GCP 무료 티어 계정 가입 및 VM 인스턴스 준비
1. 구글 클라우드 플랫폼(GCP)에 접속하여 신규 계정을 등록하고 무료 체험(300달러 크레딧) 또는 무료 티어 조건을 확인합니다.
2. Compute Engine > VM 인스턴스 메뉴에서 가상 서버 인스턴스를 하나 생성합니다.
    * OS: Ubuntu 22.04 LTS (최신 x86/64 이미지)
    * 권장 하드웨어 사양: e2-standard-4 (4 vCPU, 16GB RAM) 이상을 권장합니다.

### 2. GCP 가상 서버 방화벽(포트 개방) 규칙 구성
게임 클라이언트 접속을 허용하기 위해 인스턴스 네트워크 설정에 다음 규칙을 추가해야 합니다.
* 게임 접속 포트: UDP 8211 (소스로부터의 IP 범위: 0.0.0.0/0 - 전체 개방)
* 주의: 보안을 위해 RCON 포트(TCP 25575) 및 REST API 포트(TCP 8212)는 구글 방화벽 상에서 외부로 노출되지 않도록 폐쇄하고 로컬에서만 통신하도록 설정하십시오.

### 3. 디스코드 개발자 포털 봇 생성 및 토큰 발급
1. Discord Developer Portal(https://discord.com/developers/applications)에 접속하여 New Application을 생성합니다.
2. 왼쪽 Bot 탭으로 이동하여 봇 계정을 생성하고, Token을 안전한 곳에 복사해 둡니다. (이 값이 DISCORD_TOKEN이 됩니다.)
3. 같은 Bot 탭 화면 하단의 Privileged Gateway Intents 섹션에서 다음 세 가지 필수 권한을 활성화합니다.
    * Presence Intent
    * Server Members Intent
    * Message Content Intent
4. 왼쪽 OAuth2 > URL Generator 탭으로 이동합니다.
    * Scopes: bot, applications.commands 체크
    * Bot Permissions: Administrator (또는 채널 읽기/쓰기, 역할 제어 권한 필수 체크)
5. 하단에 자동 생성된 URL 주소를 복사하여 브라우저에 접속한 뒤, 본인이 관리하는 디스코드 서버에 봇을 초대합니다.

### 4. MobaKeyGen (또는 puttygen.exe)으로 SSH 키 생성
1. MobaXterm 상단 메뉴에서 Tools > MobaKeyGen (SSH Key Generator)을 실행합니다.
2. 하단의 Parameters 항목에서 RSA를 선택하고, Generate 버튼을 클릭합니다.
3. 진행 표시줄이 차오르는 동안 마우스를 무작위로 계속 움직여 임의의 난수를 생성합니다.
4. 키 생성이 완료되면 화면 중간의 Key comment 부분을 원하는 <사용자_계정명>으로 변경합니다. (예: gcp-user 등 영어 소문자로 구성된 계정명 권장)
5. Save private key 버튼을 클릭하여 안전한 로컬 폴더에 비공개 키 파일(.ppk)을 저장합니다.
6. 맨 위 회색 텍스트 상자(Public key for pasting into OpenSSH...) 안에 있는 긴 문자열 전체를 복사합니다.

### 5. 구글 클라우드 콘솔에 공개 키 등록
1. GCP 콘솔의 Compute Engine > VM 인스턴스 상세 페이지에서 상단의 수정 버튼을 클릭합니다.
2. 화면을 아래로 내려 SSH 키 항목을 찾은 후 항목 추가를 누릅니다.
3. 위에서 복사한 공개 키 전체를 입력란에 그대로 붙여넣습니다. (정상 등록 시 끝에 입력한 계정명이 감지됩니다.)
4. 맨 아래로 내려 저장을 누릅니다.

### 6. MobaXterm 세션 생성 및 최초 접속
1. MobaXterm 왼쪽 위 Session 버튼을 누르고 SSH 세션을 추가합니다.
2. Remote host: GCP에서 할당된 가상 서버의 외부 IP 주소를 입력합니다.
3. Specify username: 키 생성 시 Key comment에 입력했던 <사용자_계정명>을 그대로 입력합니다.
4. Advanced SSH settings 탭을 클릭한 뒤, Use private key 항목을 체크하고 저장해 두었던 비공개 키 파일(.ppk)을 등록합니다.
5. OK를 누르면 가상 서버에 암호 입력 없이 안전하게 최초 접속됩니다.

## 🛠️ 6. 단계별 구축 가이드

### [단계 1] 최초 접속 및 권한 최적화 폴더 생성

### [단계 2] 프로젝트 파일 구성 (PC에서 생성 후 업로드)
MobaXterm 왼쪽 파일 탐색 패널을 새로고침하여 생성된 `/home/<사용자_계정명>/palworld` 폴더로 진입한 후, 준비된 아래의 필수 파일들과 폴더들을 업로드합니다.

#### 💡 env 설정 파일 구성 양식 (점이 없는 파일)
env 파일은 디스코드 봇 및 경로 매핑을 위한 파일입니다. 아래 포맷 예시에 맞추어 정보를 입력하여 업로드하십시오.

    # --- 디스코드 설정 ---
    DISCORD_TOKEN=발급받은_디스코드_봇_토큰
    ADMIN_IDS=관리자_디스코드_숫자_ID (여러 명일 시 콤마로 구분 예: ID1,ID2)
    CHANNEL_ID=현황판_출력용_디스코드_채널_숫자_ID
    LOG_CHANNEL_ID=백업성공/재시작/디스크경고_로그_수신용_채널_숫자_ID

    # --- 서버 정보 ---
    SERVER_NAME=Palworld Korea
    SERVER_IP=자신의_GCP_외부_IP_주소
    SERVER_PORT=8211

    # --- 게임 내부 설정 ---
    ADMIN_PASSWORD=원하는_RCON_비밀번호 (예: 12341234)
    RCON_PORT=25575
    REST_API_PORT=8212

    # --- 경로 설정 (<사용자_계정명> 자리에는 본인의 실제 계정명을 입력하십시오) ---
    BASE_PATH=/home/<사용자_계정명>/palworld
    SAVE_DATA_PATH=/home/<사용자_계정명>/palworld/palworld/Pal/Saved
    BACKUP_PATH=/home/<사용자_계정명>/palworld/backups

    # --- 운영 정책 (숫자는 정수형으로 입력) ---
    BACKUP_RETENTION_DAYS=2                 # 백업 파일 보존 기간 (일 단위)
    MEMORY_RESTART_THRESHOLD=80             # RAM 사용량 n% 초과 시 자동 재시작
    RESTART_TIME=04:00                      # 매일 서버 자동 재시작 시간 (HH:MM 형식)

#### 💡 docker-compose.yml 파일 구성 구성
도커 컨테이너가 단순히 작동 중인 것과, 실제 게임 서버 내부 프로세스가 정상적으로 통신을 받아들일 수 있는 상태인 것은 다릅니다. 이를 정밀하게 검증하기 위해 docker-compose.yml 파일에 RCON 포트 감지 기반의 헬스체크 설정을 포함하여 구성합니다.

    version: '3.8'

    services:
      palworld:
        image: thijsvanloef/palworld-server-docker:latest
        container_name: palworld-server
        stop_grace_period: 30s
        ports:
          - 8211:8211/udp
          - 27015:27015/udp
          - 25575:25575/tcp
          - 8212:8212/tcp
        environment:
          - TZ=Asia/Seoul
          - PUID=1000
          - PGID=1000
        env_file:
          - .env
        volumes:
          - ./palworld:/palworld/

        healthcheck:
          test: ["CMD-SHELL", "timeout 5s bash -c 'cat < /dev/null > /dev/tcp/127.0.0.1/25575' || exit 1"]
          interval: 10s
          timeout: 5s
          retries: 3
          start_period: 60s

* **test (진단 명령)**: 컨테이너 내부의 로컬 루프백 주소(`127.0.0.1`)와 RCON 제어 포트(`25575`) 간의 TCP 연결이 유효한지 검사합니다. 가벼운 Bash 내장 `/dev/tcp` 채널을 통해 빈 데이터(`cat < /dev/null`)를 전송하는 방식을 사용하여 리소스 소모를 최소화합니다.
* **interval**: 컨테이너 구동 이후 매 10초 간격으로 진단 명령을 반복 수행합니다.
* **timeout**: 진단 명령을 지시한 후 5초 이내에 정상적인 TCP 응답이 도달하지 않으면 실패 처리됩니다.
* **retries**: 연속으로 3회 이상 검증에 실패할 경우, 도커 엔진은 컨테이너의 건강 상태를 `unhealthy`로 최종 격하합니다. 디스코드 관리 봇은 이 상태 값을 수집하여 현황판에 반영합니다.
* **start_period**: 컨테이너가 기동된 직후 첫 60초 동안은 상태 검사가 실패하더라도 누적 실패 횟수에서 배제함으로써, 컨테이너가 기동 중에 재시작 루프에 빠지는 현상을 방지합니다.

## 📈 7. 실시간 현황판(대시보드) 상세 메커니즘

디스코드 현황판은 시스템 상태 및 접속자 정보를 실시간으로 갱신하고 채널 유실에 자가 대응할 수 있는 지능형 구조로 작동합니다.

### 1. 통계 정보 수집 구조
현황판 정보 수집 및 GUI 갱신 기능은 `main.py` 내부의 `refresh_dashboard` 코드가 핵심 제어권을 가집니다.
* **상태 추적 파이프라인**: 봇은 `docker_service.py`를 호출하여 물리 컨테이너 상태를 검출합니다.
* **이중 단계 상태 검증**: 도커 엔진에서 반환한 컨테이너 속성 중 `Health/Status`가 `starting` 상태이면 대시보드 상태를 즉시 `STARTING`으로 매핑하고 접속자 텍스트를 "로딩 중..."으로 고정합니다.
* **실시간 성능 지표 수집**: 컨테이너 헬스체크가 정상으로 판별되면, `MonitorService` 클래스가 내부 팰월드 REST API 웹서버(`http://127.0.0.1:8212/v1/api/info` 및 `metrics`)로부터 실제 서버 버전, 현재 접속자 수, 가용한 최대 플레이어 제한 수를 수집하여 대시보드 구조에 동적으로 반영합니다.

### 2. 무중단 자가 치유(Self-Healing) 알고리즘
대시보드 채널의 대화나 메시지 삭제 작업 등으로 인해 현황판 메시지가 채널 내에서 예기치 않게 소멸되었을 경우, 시스템은 자가 치유 엔진을 통해 무중단으로 채널 구조를 정상화합니다.
* **존재 여부 검증**: `status_service.py`를 통해 디스크 상의 `status.json` 파일에서 기존 메시지 ID(`msg_id`)를 파싱합니다.
* **메시지 실시간 확인**: 디스코드 채널 API를 호출하여 해당 ID를 조회하며, 만약 삭제되어 메시지를 찾지 못하는 경우 즉각 자가 복구 스캔 프로세스를 가동합니다.
* **채널 역사 정밀 스캔**: 현황판 전용 채널의 최근 메시지 내역(최대 50개)을 샅샅이 스캔합니다. 스캔 대상 중 봇 본인의 사용자 ID가 생성한 메시지이고, 임베드의 제목 필드에 서버 이름 문자열이 부합하는 대시보드 구조를 탐색합니다.
* **바인딩 동기화 및 갱신**: 발견 시 해당 메시지 객체를 즉시 복구 지점으로 타겟팅하고 새로운 메시지 ID를 캐시 데이터 파일(`status.json`)에 등록합니다. 만약 검색 한도 내에서 대시보드 구조를 찾지 못하면 완전히 새로운 신규 현황판 메시지를 발행하여 채널 공백을 자율적으로 메웁니다.
* **이중 생성 정화 정책**: 스캔 과정 중 과거 기동 과정에서 엉켜 중복 발행된 대형 상태 임베드가 채널 내부에 누적되어 있는 것이 파악되면, 가장 최근의 대시보드 메시지 하나만 안전하게 유지한 뒤 이전 중복 메시지들은 백그라운드에서 소거(Delete) 처리하여 채널 오염을 물리적으로 차단합니다.
* **경쟁 상태 및 교착 방지 (Concurrency Protection)**: 짧은 주기에 다중 이벤트(유저 입퇴장, 시스템 주기 루틴)가 겹쳐 현황판 갱신 명령어들이 병렬로 실행될 때, 채널 상에 복수의 대시보드 메시지가 난발하는 것을 예방하고자 `self.dashboard_lock = asyncio.Lock()` 비동기 락을 구현하여 갱신 임무를 오직 단일 순차(Thread-safe)로 수행합니다.

## 🔔 8. 알림 및 로그 메시지 발송 시스템

본 시스템은 관리자가 원격 터미널에서 로그를 실시간 추적하지 않아도 모바일이나 디스크립션 봇을 통해 시스템 내부의 모든 이벤트를 추적할 수 있도록 알림 포워딩 아키텍처를 취하고 있습니다.

### 1. 채널 이원화 설계
* **대시보드 채널 (`CHANNEL_ID`)**: 현황판 대시보드 임베드와 물리 버튼 컨트롤러(`ServerControlView`)만 영구 고정되어 소통이 이루어집니다. 이 채널에 등록되는 점검 안내, 가이드 등 모든 메시지는 채널 레이아웃 청결과 대화 흐름 관리를 위해 생성 시각으로부터 **15분(900초)이 경과하면 즉시 영구 자폭 삭제** 처리됩니다.
* **백업/로그 전용 채널 (`LOG_CHANNEL_ID`)**: 현황판 채널을 오염시킬 수 있는 시스템 배후 활동(세이브 백업의 완료/실패 알림, 정기 재시작 로그, 디스크 용량 한계 임계점 경고)은 전부 이곳으로 영구 롤링 보관됩니다.

### 2. 세부 이벤트별 상황 전파 시퀀스

#### ① 정기 점검 업데이트 루틴 알림
매일 `RESTART_TIME`에 자동 점검 주기가 개시되거나 관리자의 수동 명령에 따라 점검 절차가 시작되면 아래 단계로 알림을 전송합니다.
* **점검 5분 전**: 게임 내부 전체에 RCON 방송(`Broadcast Server_will_restart_in_5_minutes`)이 울리고, 대시보드 전용 채널에 노란색 경고로 채색된 5분 전 대기 알림 메시지가 출력됩니다.
* **점검 1분 전**: 게임 내에 RCON 60초 안내(`Broadcast Server_will_restart_in_60_seconds`)가 송출되며, 채널에 점검 임박 사실을 환기하는 2차 알림 메시지가 연쇄 표출됩니다.
* **점검 30초 전**: 최종 30초 안내(`Broadcast Server_will_restart_in_30_seconds`) RCON 방송이 기재됩니다.
* **세이브 디스크 플러시**: 최종 중단에 돌입하기 전 RCON `Save` 명령을 전송하여, 활성 메모리에 누적된 세이브 데이터를 안전하게 디스크 스토리지 영역으로 밀어냅니다. 이후 10초 대기 세션을 갖고 컨테이너를 안전 정지시킨 후 업데이트 검증 및 백업 압축 루틴을 이행합니다. 점검이 완료되면 완료 알림이 1시간 동안 유지된 후 소멸됩니다.

#### ② 세이브 백업 및 자동 정리 성공/실패 영구 알림
* **백업 성공 영구 로그**: `backup_service.py`를 통해 세이브 디렉토리에 대한 타르 압축 파일(`tar.gz`) 생성이 정상 처리되면 로그 전용 채널로 파일명 규격을 투영하여 성공 메시지를 기록합니다.
* **백업 실패 긴급 알림**: 디스크 공간 포화, 경로 유실 등의 이슈로 아카이빙 시도가 거부되면 로그 전용 채널에 긴급 적색 임베드로 실패를 통보하고, 관리자 숫자 고유 ID 태그 및 구체적 원인 예외 스택 데이터를 로깅하여 즉각적인 분석을 지원합니다.
* **보관 기한 정리 메시지**: 지정된 보존 주기(`BACKUP_RETENTION_DAYS`)를 초과한 만료 백업들이 소거되면, 24시간 안전 보호 장치에 근거해 유지된 파일 내역과 디스크 상에서 안전 영구 정리된 타르 아카이브 상세 리스트를 정리하여 보고합니다.

#### ③ 서버 디스크 스페이스 초과 조기 경보
* **디스크 12시간 자가 점유율 계산**: `scheduled_disk_check_task`를 활용하여 12시간 주기로 물리 공간의 점유 한계를 진단합니다.
* **임계값 80% 초과 판정**: 드라이브 가용한 볼륨 중 점유율이 80%를 넘어서면, 로그 수신 채널로 ⚠️ [디스크 용량 경고] 적색 비상 임베드를 기동하여 상세 점유 용량 수치(`33.4 GB / 40.0 GB`)를 공표하며 신속한 보존 파일의 정리 수동 가동을 종용합니다.

#### ④ 설정값 수동 일괄 전정 이중 검증 (`/설정변경`)
* 관리자가 명령 프롬프트창을 통해 `/설정변경 EXP_RATE=2.0` 형태의 구문을 전개하면, 시스템은 env 파일 내부의 사전 키 매칭 검증을 실시하여 잘못된 오타 인자를 1차적으로 완벽 차단합니다.
* 변경의 정당성이 식별되면 이중 승인용 대화식 UI 뷰인 `ConfigConfirmView` 버튼 창을 전용 채널에 출력합니다. 이 변경 승인 패널은 오직 명령을 최초 개시한 마스터 디스코드 계정 고유 ID 세션에 의해서만 승인 버튼이 가동되며 다른 사용자의 불법 오조작을 차단합니다. 승인이 최종 승낙되면 안전 백업 완료 후 적용 재기동을 실행합니다.

## 📈 9. 실시간 운영 및 모니터링 명령어 가이드

### 1. 봇 서비스 관련 제어 명령어
디스코드 봇은 가상 서버가 켜질 때 자동으로 같이 실행되는 시스템 서비스로 등록되어 있습니다.
* 봇 상태 확인 (작동 여부, 가동 시간 등 확인):
    `sudo systemctl status palbot.service`
* 봇 재시작 (코드가 수정되었거나 env 설정값이 바뀐 경우):
    `sudo systemctl restart palbot.service`

### 2. 봇 로그 실시간 추적 (모니터링)
정기 백업 알림, 5초 주기 접속자 변경 감지 로그, 디스크 경고, 예외 발생 에러 내역을 터미널에서 모니터링할 수 있습니다.
* 실시간 로그 지켜보기 (종료하려면 Ctrl + C):
    `tail -f /home/<사용자_계정명>/palworld/palworld_bot.log`

### 3. 게임 서버 도커 컨테이너 제어 명령어
* 현재 켜져 있는 도커 컨테이너 목록 확인:
    `docker ps`
* 게임 서버 강제 정지:
    `cd ~/palworld && docker-compose down`
* 게임 서버 다시 시작:
    `cd ~/palworld && docker-compose up -d`

## 🎯 10. 유저용 디스코드 명령어 가이드

봇을 서버에 초대할 때 bot과 applications.commands 권한을 모두 체크하고 초대했다면 디스코드 대화창에서 슬래시( `/` )를 입력해 다음 관리 도구를 호출할 수 있습니다.

* **`/backup`**
    * 즉시 팰월드 게임에 저장을 명령하고, 가상 서버 로컬 드라이브에 안전하게 세이브 파일을 타르 압축하여 수동 백업을 생성합니다.
* **`/clean`**
    * 현황판 전용 채널에서 고정된 상태판 메시지 하나만 제외하고 유저들의 채팅과 시스템 메시지 목록들을 일괄 정리합니다. (사용 전, 보존할 상태판 메시지를 디스코드에서 미리 고정해야 지워지지 않습니다.)
* **`/restart`**
    * 게임 내에 셧다운 방송을 송출하고 세이브를 저장한 뒤, 안전 백업을 수행하고 도커 컨테이너를 리스타트합니다.
* **`/stop`**
    * 게임 서버를 디스코드 상에서 안전하게 정지합니다. (관리자 전용)
* **`/start`**
    * 정지된 게임 서버를 디스코드 상에서 다시 가동합니다. (관리자 전용)
* **`/디스크`**
    * 현재 서버의 디스크 전체 용량, 사용 중인 용량, 사용률(%)을 실시간으로 조회하여 임베드 메시지로 출력합니다. (관리자 전용)
* **`/설정변경 <query>`**
    * 예: `/설정변경 query:EXP_RATE=2.0, CATCH_RATE=1.5`
    * 팰월드 서버의 `.env` 환경 변수 값을 한 번에 여러 개 변경하고 서버를 재시작합니다. 변경 전 이중 승인 절차를 거칩니다. (관리자 전용)
